# RUNBOOK TITLE:
Payment Service Connection Pool Exhaustion Runbook

# INCIDENT SUMMARY:
The payment-service experienced a connection pool exhaustion, leading to a series of downstream errors and failures, including a database connection loss and a circuit breaker opening. This runbook provides steps to diagnose and resolve the issue.

# PREREQUISSES:
- Access to the payment-service application logs
- Access to the database (payments-db) logs
- Ability to update the payment-service configuration (application.properties)
- Familiarity with the payment-service codebase

# DETECTION STEPS:
1. **Check the payment-service logs** for connection pool exhaustion errors.
	* Expected outcome: Errors indicating connection pool exhaustion, such as "Connection pool exhausted" or "Unable to acquire a database connection".
2. **Check the database (payments-db) logs** for connection errors.
	* Expected outcome: Errors indicating connection issues, such as "Connection timeout" or "Connection refused".
3. **Check the circuit breaker status** to see if it has opened.
	* Expected outcome: The circuit breaker is open, indicating repeated failures in the payment-service.

# DIAGNOSIS STEPS:
1. **Check the current connection pool size** by running the following command:
```bash
grep "spring.datasource.hikari.maximum-pool-size" application.properties
```
	* Expected outcome: The current connection pool size is displayed.
2. **Check the database query performance** by running the following command:
```sql
EXPLAIN SELECT * FROM users WHERE id = ?
```
	* Expected outcome: The query execution plan is displayed, indicating potential performance bottlenecks.
3. **Check the payment-service metrics** to see if there are any issues with the connection pool or database queries.
	* Expected outcome: Metrics indicating connection pool utilization, database query performance, and other relevant metrics are displayed.

# RESOLUTION STEPS:
1. **Increase the connection pool size** by updating the `application.properties` file:
```properties
spring.datasource.hikari.maximum-pool-size=20
```
	* Expected outcome: The connection pool size is increased, allowing more database connections to be established.
2. **Implement a retry mechanism** by adding the following code to the payment-service:
```java
@Service
public class PaymentService {
    
    @Retryable(value = {SQLException.class}, maxAttempts = 3, backoff = @Backoff(delay = 500))
    public void makePayment(Payment payment) {
        // database operation
    }
}
```
	* Expected outcome: The retry mechanism is implemented, allowing the payment-service to retry failed database operations.
3. **Optimize database queries** by analyzing and optimizing database queries to reduce execution time.
	* Expected outcome: Database queries are optimized, reducing the load on the database and improving performance.

# VERIFICATION STEPS:
1. **Check the payment-service logs** to see if the connection pool exhaustion errors have stopped.
	* Expected outcome: The connection pool exhaustion errors have stopped, indicating that the issue has been resolved.
2. **Check the database (payments-db) logs** to see if the connection errors have stopped.
	* Expected outcome: The connection errors have stopped, indicating that the issue has been resolved.
3. **Check the circuit breaker status** to see if it has closed.
	* Expected outcome: The circuit breaker is closed, indicating that the payment-service is functioning normally.

# ROLLBACK PLAN:
If the changes made to resolve the issue cause further problems, the following steps can be taken to roll back:
1. **Revert the connection pool size change** by updating the `application.properties` file to its original value.
2. **Remove the retry mechanism** by removing the `@Retryable` annotation from the payment-service code.
3. **Revert any database query optimizations** by reverting the changes made to the database queries.

# ESCALATION PATH:
If the issue is not resolved after following the steps in this runbook, the following escalation path can be followed:
1. **Notify the on-call engineer** to investigate the issue further.
2. **Notify the development team** to provide additional support and guidance.
3. **Notify the operations team** to provide additional support and guidance.

# PREVENTION CHECKLIST:
To prevent similar issues in the future, the following checklist can be followed:
1. **Monitor connection pool metrics** to detect potential issues before they occur.
2. **Optimize database queries** to reduce the load on the database and improve performance.
3. **Implement a circuit breaker pattern** to detect and prevent cascading failures.
4. **Implement a queue-based architecture** to handle high volumes of requests.
5. **Regularly review and update the payment-service configuration** to ensure it is optimized for performance and reliability.
