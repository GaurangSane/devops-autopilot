# RUNBOOK TITLE:
PostgreSQL Database Connection Failure Runbook

# INCIDENT SUMMARY:
The PostgreSQL database connection failed, causing the API to crash. The error occurred when trying to establish a connection to the database server at `127.0.0.1:5432`. This runbook provides step-by-step instructions to diagnose and resolve the issue.

# PREREQUISSES:
- Access to the server hosting the PostgreSQL database
- Sudo privileges to start and stop the PostgreSQL service
- Ability to edit configuration files (e.g., `postgresql.conf`)

# DETECTION STEPS:
1. **Check API logs**: Look for error messages indicating a database connection failure.
2. **Verify PostgreSQL server status**: Run the command `sudo service postgresql status` to check if the PostgreSQL server is running.
3. **Check for recent changes**: Review recent configuration changes or updates that may have affected the PostgreSQL server.

# DIAGNOSIS STEPS:
1. **Check `listen_addresses` in `postgresql.conf`**: Verify that PostgreSQL is configured to listen on the correct IP address.
	* Command: `sudo nano /etc/postgresql/common/postgresql.conf` (path may vary)
	* Check the value of `listen_addresses` and ensure it includes `127.0.0.1` or `*` for all available interfaces.
2. **Check PostgreSQL server logs**: Inspect the PostgreSQL server logs for any error messages indicating why the server is not accepting connections.
	* Command: `sudo nano /var/log/postgresql/postgres.log` (path may vary)
3. **Verify firewall rules**: Ensure that the firewall is not blocking connections to the PostgreSQL server.
	* Command: `sudo ufw status` (for Ubuntu-based systems)

# RESOLUTION STEPS:
1. **Start PostgreSQL server if not running**: If the server is not running, start it.
	* Command: `sudo service postgresql start`
2. **Adjust `listen_addresses` in `postgresql.conf` if necessary**: If the `listen_addresses` value is incorrect, update it to include `127.0.0.1` or `*`.
	* Command: `sudo nano /etc/postgresql/common/postgresql.conf` (path may vary)
	* Example config change: `listen_addresses = '127.0.0.1'` or `listen_addresses = '*'`
3. **Restart PostgreSQL server after configuration changes**: Apply the changes made to `postgresql.conf`.
	* Command: `sudo service postgresql restart`
4. **Implement database connection retry mechanism**: Modify the application to retry database connections upon failure.
	* Example using `psycopg2`:
	```python
import psycopg2
from psycopg2 import OperationalError

def connect_with_retry(db_config, retry_count=3):
    for attempt in range(retry_count):
        try:
            return psycopg2.connect(**db_config)
        except OperationalError as e:
            if attempt < retry_count - 1:
                print(f"Connection attempt {attempt+1} failed: {e}. Retrying...")
                time.sleep(1)  # Wait before retrying
            else:
                raise
```

# VERIFICATION STEPS:
1. **Check API logs**: Verify that the API is no longer reporting database connection errors.
2. **Verify PostgreSQL server status**: Run the command `sudo service postgresql status` to ensure the server is running and accepting connections.
3. **Test database connection**: Use a tool like `psql` to test the database connection.
	* Command: `psql -h 127.0.0.1 -p 5432 -U postgres`

# ROLLBACK PLAN:
If the changes made to resolve the issue cause further problems:
1. **Revert `listen_addresses` change**: Update `postgresql.conf` to its original value.
	* Command: `sudo nano /etc/postgresql/common/postgresql.conf` (path may vary)
2. **Restart PostgreSQL server**: Apply the changes made to `postgresql.conf`.
	* Command: `sudo service postgresql restart`
3. **Revert database connection retry mechanism**: Remove the retry mechanism implemented in the application.

# ESCALATION PATH:
If the issue cannot be resolved using this runbook:
1. **Notify the database administrator**: Inform the database administrator of the issue and request assistance.
2. **Notify the development team**: Inform the development team of the issue and request assistance with implementing a database connection retry mechanism.
3. **Escalate to incident management**: If the issue persists, escalate to incident management for further assistance.

# PREVENTION CHECKLIST:
To prevent similar issues in the future:
1. **Regularly review PostgreSQL server logs**: Inspect the PostgreSQL server logs for any error messages indicating potential issues.
2. **Monitor PostgreSQL server status**: Use a monitoring tool to track the PostgreSQL server's status and receive alerts if the server is not running or not accepting connections.
3. **Implement a comprehensive monitoring and alerting system**: Set up a monitoring system to detect issues before they become incidents.
4. **Regularly review and update `postgresql.conf`**: Ensure that the `postgresql.conf` file is up-to-date and correctly configured.
5. **Test database connection retry mechanism**: Regularly test the database connection retry mechanism to ensure it is functioning correctly.
