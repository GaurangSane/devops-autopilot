# RUNBOOK TITLE:
Payment Service Connection Pool Exhaustion Runbook

# INCIDENT SUMMARY:
The payment service experienced a connection pool exhaustion, leading to a series of failures including transaction rollbacks, payment verification timeouts, and ultimately a loss of database connection. This incident resulted in a complete loss of functionality for the payment service, leading to failed orders and a compensation workflow being initiated.

# PREREQUISSES:
- Access to the payment service configuration files
- Access to the database (postgresql) configuration files
- Ability to execute SQL queries on the database
- Familiarity with the payment service and its dependencies (api-gateway, order-service, load-balancer)

# DETECTION STEPS:
1. **Check the payment service logs** for connection pool exhaustion errors.
	* Expected outcome: Errors indicating connection pool exhaustion, such as "Connection pool exhausted" or "Unable to acquire new connection".
2. **Verify the database connection status** using SQL queries.
	* Command: `psql -U <username> -d <database> -c "SELECT * FROM pg_stat_activity;"`
	* Expected outcome: A list of active database connections, including the payment service connections.
3. **Check the load balancer and api-gateway logs** for errors related to the payment service.
	* Expected outcome: Errors indicating that the payment service is unavailable, such as "503 Service Unavailable" or "Payment service timeout".

# DIAGNOSIS STEPS:
1. **Check the connection pool configuration** to determine if the maximum pool size is set too low.
	* Command: `grep maximumPoolSize /path/to/payment/service/config.properties`
	* Expected outcome: The current maximum pool size value, such as `maximumPoolSize=20`.
2. **Verify the idle timeout setting** to determine if it is set too high.
	* Command: `grep idleTimeout /path/to/payment/service/config.properties`
	* Expected outcome: The current idle timeout value, such as `idleTimeout=300000`.
3. **Check the database query performance** to determine if it is causing excessive load on the database.
	* Command: `psql -U <username> -d <database> -c "EXPLAIN (ANALYZE) <query>;"`
	* Expected outcome: The query execution plan, including the estimated cost and time.

# RESOLUTION STEPS:
1. **Increase the maximum pool size** to prevent connection pool exhaustion.
	* Command: `sed -i 's/maximumPoolSize=20/maximumPoolSize=50/' /path/to/payment/service/config.properties`
	* Expected outcome: The maximum pool size is increased to 50.
2. **Adjust the idle timeout** to prevent connections from remaining idle for too long.
	* Command: `sed -i 's/idleTimeout=300000/idleTimeout=30000/' /path/to/payment/service/config.properties`
	* Expected outcome: The idle timeout is reduced to 30 seconds.
3. **Set the connection timeout** to prevent requests from waiting indefinitely for a connection.
	* Command: `sed -i 's/connectionTimeout=0/connectionTimeout=30000/' /path/to/payment/service/config.properties`
	* Expected outcome: The connection timeout is set to 30 seconds.
4. **Restart the payment service** to apply the configuration changes.
	* Command: `systemctl restart payment-service`
	* Expected outcome: The payment service is restarted and the connection pool is reinitialized.

# VERIFICATION STEPS:
1. **Check the payment service logs** for errors related to connection pool exhaustion.
	* Expected outcome: No errors indicating connection pool exhaustion.
2. **Verify the database connection status** using SQL queries.
	* Command: `psql -U <username> -d <database> -c "SELECT * FROM pg_stat_activity;"`
	* Expected outcome: A list of active database connections, including the payment service connections.
3. **Check the load balancer and api-gateway logs** for errors related to the payment service.
	* Expected outcome: No errors indicating that the payment service is unavailable.

# ROLLBACK PLAN:
1. **Revert the configuration changes** to the previous values.
	* Command: `sed -i 's/maximumPoolSize=50/maximumPoolSize=20/' /path/to/payment/service/config.properties`
	* Expected outcome: The maximum pool size is reverted to 20.
2. **Restart the payment service** to apply the configuration changes.
	* Command: `systemctl restart payment-service`
	* Expected outcome: The payment service is restarted and the connection pool is reinitialized.

# ESCALATION PATH:
1. **Notify the on-call engineer** if the issue persists after applying the resolution steps.
2. **Escalate to the database team** if the issue is related to the database configuration or performance.
3. **Escalate to the development team** if the issue is related to the payment service code or configuration.

# PREVENTION CHECKLIST:
1. **Monitor the connection pool metrics** to detect potential issues before they become critical.
2. **Regularly review the database query performance** to optimize queries and reduce load on the database.
3. **Implement retry mechanisms and circuit breakers** to handle temporary database connection failures and prevent cascading failures.
4. **Optimize the payment service configuration** to prevent connection pool exhaustion and improve overall performance.
