# RUNBOOK TITLE:
Payment Service Connection Pool Exhaustion Runbook

# INCIDENT SUMMARY:
The payment-service experienced a connection pool exhaustion, leading to a series of failures including transaction rollbacks, payment verification timeouts, and ultimately a loss of database connection. This incident resulted in failed orders, transaction rollbacks, and the payment-service being marked as unhealthy.

# PREREQUISITES:
- Access to the payment-service configuration files (e.g., `application.properties`)
- Ability to execute commands on the payment-service server
- Access to the database (payments-db) for query optimization and indexing
- Knowledge of the system's architecture and dependencies

# DETECTION STEPS:
1. **Monitor payment-service logs**: Check the payment-service logs for connection pool exhaustion errors, transaction rollbacks, and payment verification timeouts.
	* Expected outcome: Identify the error patterns and timestamps.
2. **Check service health**: Verify the payment-service health check status.
	* Expected outcome: Confirm if the service is marked as unhealthy.
3. **Review database performance metrics**: Check the database performance metrics (e.g., connection utilization, query latency).
	* Expected outcome: Identify any performance issues or bottlenecks.

# DIAGNOSIS STEPS:
1. **Verify connection pool configuration**: Check the `spring.datasource.hikari.maximum-pool-size` property in the `application.properties` file.
	* Expected outcome: Confirm if the connection pool size is adequate for the current demand.
2. **Analyze database queries**: Use MySQL’s EXPLAIN statement to analyze query execution plans and identify slow queries.
	* Expected outcome: Identify queries that need optimization.
3. **Check indexing strategies**: Evaluate the indexing strategies for columns frequently used in search conditions and JOIN conditions.
	* Expected outcome: Identify columns that need indexing.

# RESOLUTION STEPS:
1. **Increase connection pool size**: Edit the `application.properties` file and update `spring.datasource.hikari.maximum-pool-size=50`.
	* Expected outcome: The connection pool size is increased to handle the current demand.
2. **Implement connection pooling with R2DBC**: Add the following properties to the `application.properties` file:
	* `spring.r2dbc.pool.enabled=true`
	* `spring.r2dbc.pool.initial-size=50`
	* `spring.r2dbc.pool.max-size=100`
	* Expected outcome: Connection pooling is enabled for R2DBC.
3. **Optimize slow queries**: Use MySQL’s EXPLAIN statement to analyze query execution plans and optimize slow queries.
	* Expected outcome: Slow queries are optimized to minimize unnecessary computations and leverage indexes effectively.
4. **Implement proper indexing**: Evaluate and adjust indexing strategies to ensure columns frequently used in search conditions and JOIN conditions are indexed.
	* Expected outcome: Indexing strategies are optimized to improve query performance.

# VERIFICATION STEPS:
1. **Monitor payment-service logs**: Check the payment-service logs for connection pool exhaustion errors, transaction rollbacks, and payment verification timeouts.
	* Expected outcome: The error patterns and timestamps are resolved.
2. **Check service health**: Verify the payment-service health check status.
	* Expected outcome: The service is marked as healthy.
3. **Review database performance metrics**: Check the database performance metrics (e.g., connection utilization, query latency).
	* Expected outcome: The performance issues or bottlenecks are resolved.

# ROLLBACK PLAN:
1. **Revert connection pool size**: Edit the `application.properties` file and update `spring.datasource.hikari.maximum-pool-size` to its original value.
2. **Disable connection pooling with R2DBC**: Remove the R2DBC connection pooling properties from the `application.properties` file.
3. **Revert query optimizations**: Revert the optimized queries to their original state.
4. **Revert indexing changes**: Revert the indexing strategies to their original state.

# ESCALATION PATH:
- If the issue persists after implementing the resolution steps, escalate to the database team for further assistance.
- If the issue is related to the database hardware, escalate to the infrastructure team for further assistance.
- If the issue is related to the application code, escalate to the development team for further assistance.

# PREVENTION CHECKLIST:
- Regularly monitor connection pool utilization and database performance metrics.
- Implement proactive monitoring and alerting for connection pool exhaustion and database performance issues.
- Regularly review and optimize database queries and indexing strategies.
- Consider implementing database redundancy and failover capabilities to ensure high availability.
- Schedule regular index maintenance tasks to prevent index degradation.
- Evaluate and adjust the connection pool size and database hardware as needed to handle changing demand.
