# RUNBOOK TITLE:
PostgreSQL Database Connection Failure Runbook

# INCIDENT SUMMARY:
The PostgreSQL database connection failed, causing the API to crash. The `psycopg2` library threw an `OperationalError` exception when attempting to connect to the PostgreSQL server at `127.0.0.1:5432`.

# PREREQUISITES:
* Access to the PostgreSQL server and its configuration files
* Ability to restart the PostgreSQL server
* Familiarity with the `psycopg2` library and the API's database connection code

# DETECTION STEPS:
1. Check the API logs for errors related to database connections.
2. Verify that the PostgreSQL server is running by executing the command `sudo service postgresql status`.
3. Check the PostgreSQL server's configuration files to ensure it is listening on the correct port (default: 5432) by executing the command `grep -H '^port' /etc/postgresql//main/postgresql.conf`.

# DIAGNOSIS STEPS:
1. Check if the PostgreSQL server is running and listening on the correct port.
2. Verify that the `psycopg2` library is correctly configured and imported in the API's database connection code.
3. Check the API's database connection code for any errors or exceptions that may be causing the connection failure.
4. IF the PostgreSQL server is not running, THEN proceed to the RESOLUTION STEPS to start the server.
5. IF the PostgreSQL server is running but not listening on the correct port, THEN proceed to the RESOLUTION STEPS to update the configuration files.

# RESOLUTION STEPS:
1. Start the PostgreSQL server by executing the command `sudo service postgresql start`.
2. Check the PostgreSQL configuration files to ensure it is listening on the correct port (default: 5432) by executing the command `grep -H '^port' /etc/postgresql//main/postgresql.conf`.
3. IF the port is not set to 5432, THEN edit the `postgresql.conf` file to set the port to 5432.
4. Restart the PostgreSQL server by executing the command `sudo service postgresql restart`.
5. Verify that the API can connect to the PostgreSQL server by checking the API logs for successful database connections.

# VERIFICATION STEPS:
1. Check the API logs for successful database connections.
2. Verify that the PostgreSQL server is running and listening on the correct port by executing the command `sudo service postgresql status`.
3. Test the API's functionality to ensure it is working as expected.

# ROLLBACK PLAN:
1. IF the changes made to the PostgreSQL configuration files cause issues, THEN revert the changes by editing the `postgresql.conf` file to its original state.
2. IF the restart of the PostgreSQL server causes issues, THEN try to restart the server again or seek assistance from a senior engineer.

# ESCALATION PATH:
1. IF the issue is not resolved after following the RESOLUTION STEPS, THEN escalate the issue to a senior engineer for further assistance.
2. IF the issue is causing significant downtime or data loss, THEN escalate the issue to the on-call engineer or the incident response team.

# PREVENTION CHECKLIST:
1. Regularly check the PostgreSQL server's configuration files to ensure it is listening on the correct port.
2. Implement retry logic for database connections using a library like `tenacity`.
3. Configure PostgreSQL to accept connections from any IP by editing the `postgresql.conf` file.
4. Update the `pg_hba.conf` file to trust connections from any IP.
5. Implement a connection pool using a library like `psycopg2` or `pgbouncer`.
6. Monitor PostgreSQL server performance using tools like `pg_stat_statements` or `Prometheus`.
7. Implement idempotent database operations to allow for safe retries in case of failures.
