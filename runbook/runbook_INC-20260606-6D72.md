# RUNBOOK TITLE:
Payment Service Connection Pool Exhaustion Runbook

# INCIDENT SUMMARY:
The payment-service has experienced a connection pool exhaustion, leading to downstream errors and failures, including database connection loss and health check failure. This incident has resulted in failed orders and the initiation of a compensation workflow.

# PREREQUISITES:
- Access to the payment-service configuration files
- Access to the Kubernetes cluster
- Familiarity with HikariCP configuration
- Familiarity with Kubernetes' Horizontal Pod Autoscaler

# DETECTION STEPS:
1. **Monitor payment-service logs**: Check the payment-service logs for connection pool exhaustion errors.
	* Expected outcome: Identify the error message indicating connection pool exhaustion.
2. **Check database connection status**: Verify the database connection status using the following command:
	```bash
	psql -h <database-host> -U <database-username> -d <database-name> -c "SELECT 1;"
	```
	* Expected outcome: Successful connection to the database or an error message indicating connection failure.
3. **Check health check status**: Verify the health check status of the payment-service using the following command:
	```bash
	curl -X GET http://<payment-service-host>:<payment-service-port>/health
	```
	* Expected outcome: A successful response indicating the service is healthy or an error message indicating the service is not healthy.

# DIAGNOSIS STEPS:
1. **Check connection pool size**: Verify the current connection pool size using the following command:
	```bash
	grep "spring.datasource.hikari.maximum-pool-size" application.properties
	```
	* Expected outcome: The current connection pool size.
2. **Check database query performance**: Analyze database query performance using the following command:
	```bash
	psql -h <database-host> -U <database-username> -d <database-name> -c "EXPLAIN ANALYZE <query>;"
	```
	* Expected outcome: Query performance metrics.
3. **Check system load**: Verify the system load using the following command:
	```bash
	top -b -n 1
	```
	* Expected outcome: Current system load metrics.

# RESOLUTION STEPS:
1. **Increase connection pool size**: Update the `application.properties` file with the following configuration:
	```
	spring.datasource.hikari.maximum-pool-size=50
	spring.datasource.hikari.minimum-idle=10
	spring.datasource.hikari.idle-timeout=300000
	spring.datasource.hikari.connection-timeout=30000
	```
	* Expected outcome: Increased connection pool size.
2. **Implement connection leak detection**: Add connection leak detection to the `application.properties` file:
	```
	spring.datasource.hikari.leak-detection-threshold=60000
	```
	* Expected outcome: Connection leak detection enabled.
3. **Implement auto-scaling**: Create a `HorizontalPodAutoscaler` resource with the following configuration:
	```
	apiVersion: autoscaling/v2
	kind: HorizontalPodAutoscaler
	metadata:
	  name: payment-service-hpa
	spec:
	  selector:
	    matchLabels:
	      app: payment-service
	  minReplicas: 1
	  maxReplicas: 10
	  metrics:
	  - type: Resource
	    resource:
	      name: cpu
	      target:
	        type: Utilization
	        averageUtilization: 60
	```
	* Expected outcome: Auto-scaling enabled for the payment-service.

# VERIFICATION STEPS:
1. **Verify connection pool size**: Check the connection pool size using the following command:
	```bash
	grep "spring.datasource.hikari.maximum-pool-size" application.properties
	```
	* Expected outcome: Updated connection pool size.
2. **Verify database connection status**: Verify the database connection status using the following command:
	```bash
	psql -h <database-host> -U <database-username> -d <database-name> -c "SELECT 1;"
	```
	* Expected outcome: Successful connection to the database.
3. **Verify health check status**: Verify the health check status of the payment-service using the following command:
	```bash
	curl -X GET http://<payment-service-host>:<payment-service-port>/health
	```
	* Expected outcome: A successful response indicating the service is healthy.

# ROLLBACK PLAN:
1. **Revert connection pool size**: Update the `application.properties` file with the original connection pool size configuration.
2. **Disable connection leak detection**: Remove the connection leak detection configuration from the `application.properties` file.
3. **Disable auto-scaling**: Delete the `HorizontalPodAutoscaler` resource.

# ESCALATION PATH:
- If the issue persists after increasing the connection pool size and implementing connection leak detection, escalate to the on-call engineer.
- If the issue is related to database query performance, escalate to the database administrator.
- If the issue is related to system load, escalate to the system administrator.

# PREVENTION CHECKLIST:
- Regularly review connection pool size and adjust as needed.
- Implement connection leak detection and monitoring.
- Regularly review database query performance and optimize as needed.
- Implement auto-scaling for the payment-service.
- Regularly review system load and adjust resources as needed.
- Implement load testing to identify and mitigate bottlenecks.
- Implement the circuit breaker pattern to prevent cascading failures.
