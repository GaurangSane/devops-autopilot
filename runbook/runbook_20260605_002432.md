# RUNBOOK TITLE:
Payment-Service Connection Pool Exhaustion Runbook

# INCIDENT SUMMARY:
The payment-service experienced a connection pool exhaustion, leading to a cascade of failures including transaction rollbacks, payment verification timeouts, and ultimately a loss of database connection. This incident resulted in a complete loss of functionality for the payment-service, leading to failed orders and a compensation workflow being initiated.

# PREREQUISITES:
- Access to the payment-service configuration files (e.g., `application.properties`)
- Access to the payment-service codebase
- Access to the database configuration (e.g., PostgreSQL)
- Familiarity with the payment-service architecture and dependencies
- Basic understanding of Java and Spring Boot

# DETECTION STEPS:
1. **Monitor logs**: Check the payment-service logs for connection pool exhaustion errors, failed to acquire connection timeouts, transaction rollback errors, payment verification timeouts, and database connection loss errors.
2. **Check database connection metrics**: Monitor database connection metrics (e.g., connection pool size, active connections, idle connections) using a monitoring tool such as Prometheus and Grafana.
3. **Verify service status**: Check the status of the payment-service and dependent services (e.g., api-gateway, order-service) using a service monitoring tool.

# DIAGNOSIS STEPS:
1. **Verify connection pool configuration**: Check the connection pool configuration (e.g., `spring.datasource.hikari.maximum-pool-size`, `spring.datasource.hikari.minimum-idle`) to ensure it is adequate for the current demand.
2. **Analyze transaction handling**: Review the transaction handling code to identify potential inefficiencies or bottlenecks that may be contributing to connection pool exhaustion.
3. **Check for retry mechanism**: Verify if a retry mechanism with a circuit breaker pattern is implemented to prevent cascading failures.
4. **Review database performance**: Check database performance metrics (e.g., query latency, connection utilization) to identify potential issues.

# RESOLUTION STEPS:
1. **Increase connection pool size**:
	* Update the `application.properties` file with the following configuration:
		```
		spring.datasource.hikari.maximum-pool-size=50
		spring.datasource.hikari.minimum-idle=10
		```
	* Restart the payment-service.
	* Expected outcome: Connection pool size increased, and the service can handle more concurrent requests.
2. **Implement retry mechanism**:
	* Add the Netflix Hystrix dependency to the `pom.xml` file (if using Maven):
		```
		<dependency>
			<groupId>com.netflix.hystrix</groupId>
			<artifactId>hystrix-core</artifactId>
			<version>1.5.18</version>
		</dependency>
		```
	* Configure Hystrix to wrap the database calls. For example:
		```
		@HystrixCommand(fallbackMethod = "fallbackMethod")
		public String fetchData() {
			// Database call
		}
		
		public String fallbackMethod() {
			// Fallback logic
		}
		```
	* Restart the payment-service.
	* Expected outcome: Retry mechanism implemented, and the service can recover from temporary failures.
3. **Optimize transaction handling**:
	* Review the transaction handling code and optimize it by implementing efficient transaction management. For example:
		```
		@Transactional(readOnly = true)
		public List<Data> fetchData() {
			// Read-only transaction
		}
		```
	* Restart the payment-service.
	* Expected outcome: Transaction handling optimized, and connection pool utilization improved.

# VERIFICATION STEPS:
1. **Monitor logs**: Verify that the connection pool exhaustion errors, failed to acquire connection timeouts, transaction rollback errors, payment verification timeouts, and database connection loss errors are no longer present in the logs.
2. **Check database connection metrics**: Verify that the database connection metrics (e.g., connection pool size, active connections, idle connections) are within expected ranges.
3. **Verify service status**: Check the status of the payment-service and dependent services (e.g., api-gateway, order-service) to ensure they are functioning correctly.

# ROLLBACK PLAN:
1. **Revert connection pool size**: If the increased connection pool size causes performance issues, revert the changes to the original configuration.
2. **Disable retry mechanism**: If the retry mechanism causes issues, disable it and revert to the original configuration.
3. **Revert transaction handling changes**: If the optimized transaction handling causes issues, revert the changes to the original configuration.

# ESCALATION PATH:
1. **Notify the on-call engineer**: If the issue persists after attempting the resolution steps, notify the on-call engineer for further assistance.
2. **Escalate to the development team**: If the on-call engineer is unable to resolve the issue, escalate it to the development team for further investigation and resolution.
3. **Notify stakeholders**: If the issue has a significant impact on the business, notify stakeholders (e.g., product managers, customer support) to ensure they are aware of the situation.

# PREVENTION CHECKLIST:
1. **Regularly review connection pool configuration**: Ensure the connection pool configuration is adequate for the current demand and adjust as needed.
2. **Monitor database performance**: Regularly monitor database performance metrics to identify potential issues.
3. **Implement connection pool monitoring**: Implement connection pool monitoring to detect early signs of connection pool depletion.
4. **Regularly review transaction handling**: Regularly review transaction handling code to ensure it is optimized and efficient.
5. **Implement load balancing and auto-scaling**: Consider implementing load balancing and auto-scaling to distribute traffic and adjust to changing demand.
